# Codex 编码行为规范

## 目标
- 让 Codex 生成的代码更容易接手：关键意图有注释，局部实现不过度拆碎。
- 先复用当前模块结构；新增文件、抽象、helper 必须有明确职责和调用方。

## 注释规范
- 注释解释“为什么这样做”，不要复述代码“做了什么”。
- 以下场景必须写简短注释或 docstring：
  - 业务规则、农事/账务/Agent 路由等领域决策不容易从代码名看出原因。
  - 兼容旧入口、历史数据格式、迁移窗口或临时兜底逻辑。
  - 缓存、并发、重试、幂等、限流、权限、安全边界等非显然约束。
  - 第三方接口、LLM 输出、工具调用协议存在格式陷阱或异常分支。
  - 测试中构造数据的业务含义会影响断言理解。
- Python 对跨模块调用的 public function/class 写 docstring；私有函数只在逻辑不显然时注释。
- TypeScript/React 对复杂 hook、状态同步、数据映射和副作用边界写注释；普通 JSX 不写装饰性注释。
- 删除失效注释；禁止留下 TODO/FIXME 代替实现或解释。

## 文件拆分规范
- 默认在现有文件或同目录现有模块中完成局部改动。
- 新增文件前必须满足至少一个条件：
  - 该文件代表清晰业务边界，如 use case、repository、adapter、schema、component、hook。
  - 当前文件已接近限制：Python 500 行、React 页面 300 行、组件 150 行。
  - 同一逻辑已被 3 个及以上调用方复用，抽出后能减少真实重复。
  - 需要隔离 IO/协议适配/外部依赖，方便测试和 mock。
- 禁止创建只有一个小函数、一个常量、一个类型别名的碎片文件，除非它是既有架构约定的一部分。
- 页面或 use case 的私有 helper 优先留在本文件底部；只有跨文件复用或独立测试价值明确时才移动。
- 新增抽象必须在提交说明或实现说明中写明：调用方、边界、为什么不能留在原文件。

## 实施检查
- 写代码前先说明本次是否需要新增文件；不需要时明确“保持在现有模块内”。
- 每次新增文件后检查：是否有测试覆盖、是否被实际引用、是否符合模块边界。
- Review 时优先拦截两类问题：关键业务无注释、无必要的小文件/小抽象。
