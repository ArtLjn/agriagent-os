# 设计图到代码交接清单

> 目标：解决"设计图好看，但代码实现不了或实现后变形"的问题。任何 UI 图进入 Codex 实现前，都必须先通过这份交接。

## 为什么设计图会落不了地

常见原因不是 Codex 不会写样式，而是设计图缺少工程信息：

- 只展示理想数据，没有长文本、空数据、接口失败。
- 没有说明哪些是固定尺寸，哪些要自适应。
- 组件边界不清，Codex 无法判断该拆几个组件。
- 色值、圆角、阴影、字号没有映射到现有 theme token。
- 动效、毛玻璃、渐变、图表等效果没有说明 RN 是否可实现。
- 设计稿没有考虑键盘、安全区、低端 Android 性能。
- 截图只是一屏，没有滚动区域和状态流转。

## 交接产物

每张进入实现的 UI 图，都要配一份交接说明。最少包含：

1. 页面结构拆解
2. 组件清单
3. token 映射
4. 数据和状态
5. 交互行为
6. 技术可行性
7. 降级方案
8. 验收截图标准

## 1. 页面结构拆解

把设计图拆成从外到内的结构，避免 Codex 直接照图片堆样式。

```text
Screen
- SafeArea
- Header
  - BackButton
  - Title
  - RightAction
- ScrollContent
  - HeroSummary
  - FormSection
  - SuggestionPanel
  - HistoryList
- BottomActionBar
```

必须说明：

- 哪个区域固定在顶部
- 哪个区域跟随滚动
- 哪个区域固定在底部
- 哪个区域允许横向滚动
- 哪个区域在键盘弹出后需要避让

## 2. 组件清单

每个组件都要说明输入、状态和尺寸策略。

| 组件 | 输入数据 | 状态 | 尺寸策略 | 备注 |
|------|---------|------|---------|------|
| Header | title, rightAction | default | 固定高度 | 遵循安全区 |
| SummaryCard | amount, count, trend | loading/empty/default | 宽度撑满 | 长金额缩小字号 |
| CategoryPicker | categories, selectedId | loading/empty/error/default | 网格自适应 | 不超过两行时避免滚动 |

组件拆分原则：

- 复用超过 2 次才抽通用组件。
- 单一页面专用组件放在页面 `components/` 目录。
- 只服务本次页面的样式不要放进全局 theme。
- 已稳定的颜色、圆角、字号才进入 `src/theme`。

## 3. Token 映射

设计图里的每个视觉属性都要映射到现有 token。不要让 Codex临时造颜色。

| 设计项 | 设计值 | 代码 token | 是否允许新增 |
|-------|-------|------------|-------------|
| 主按钮背景 | 蓝色渐变 | `gradients.primary` | 否 |
| 页面背景 | 浅灰蓝 | `colors.background` | 否 |
| 卡片圆角 | 18 | `radii.lg` | 否 |
| 标题字号 | 22 | `typography.heading2` | 否 |
| 卡片阴影 | 柔和蓝阴影 | `shadowV2.card` | 否 |

如果设计值找不到 token，先判断：

1. 是否可以用相近 token 替代。
2. 是否只有这一个页面需要。
3. 是否会破坏 `docs/ui/app/ui.md` 的整体风格。

只有满足长期复用价值，才新增 token。

## 4. 数据和状态

设计图必须补齐真实数据样例。

### 默认数据

```json
{
  "amount": 1280.5,
  "category": "肥料",
  "note": "复合肥 4 袋，含运输费",
  "date": "2026-06-06"
}
```

### 边界数据

```json
{
  "amount": 9999999.99,
  "category": "很长很长的农资分类名称",
  "note": "一段超过两行的备注，用来检查卡片高度、列表换行和按钮是否被挤压",
  "date": "2026-12-31"
}
```

必须覆盖：

- loading
- empty
- error
- long text
- disabled
- submitting
- offline 或 weak network

## 5. 交互行为

不要只描述静态画面，要说明每个操作发生什么。

| 操作 | 触发 | 反馈 | 成功结果 | 失败结果 |
|------|------|------|----------|----------|
| 保存 | 点击主按钮 | 按钮 loading + 禁用 | 返回列表并刷新 | 显示错误提示，保留输入 |
| 选择分类 | 点击分类项 | pressed 状态 | 更新选中态 | 不适用 |
| 删除 | 点击删除按钮 | 二次确认 | 删除后返回 | 显示错误提示 |

移动端表单必须说明：

- 键盘弹出后是否滚动到当前输入框
- 提交按钮是否固定底部
- 输入框之间如何跳转
- 金额、日期、备注使用什么键盘类型

## 6. 技术可行性检查

实现前逐项判断：

- [ ] React Native 是否支持该效果。
- [ ] 是否已有项目依赖可实现。
- [ ] 是否需要新增依赖。
- [ ] Android 是否支持。
- [ ] 低端设备是否会卡顿。
- [ ] 是否影响可访问性。
- [ ] 是否可以截图验收。

高风险效果处理：

| 效果 | 风险 | 默认策略 |
|------|------|----------|
| 毛玻璃 | Android 兼容不稳定 | 半透明背景 + 边框降级 |
| 复杂粒子动画 | 低端机卡顿 | 不做，改为轻量 Animated |
| 大面积渐变 | 容易抢内容 | 仅用于主按钮、天气等业务强调 |
| 渐变文字 | RN 实现复杂 | 优先纯色或小范围使用 |
| 复杂图表 | 依赖和性能风险 | 先用简化条形/折线 |

## 7. 降级方案

每个高风险视觉效果都要有降级方案。

示例：

```text
设计效果：底部导航毛玻璃。
首选实现：iOS 使用 BlurView。
Android 降级：rgba(255,255,255,0.86) + borderTopWidth + shadowV2.light。
不可接受：为了毛玻璃新增重型依赖或导致 Android 滚动卡顿。
```

## 8. 验收截图标准

每张设计图至少对应以下截图：

- 设计参考图
- 实现默认状态
- 实现长文本状态
- 实现 loading 或 submitting 状态
- 实现 error 或 empty 状态
- 键盘弹出状态
- 小屏 Android 状态

验收不是比较"像不像"，而是比较：

- 信息层级是否一致
- 关键尺寸比例是否一致
- 组件边界是否一致
- token 是否一致
- 状态是否完整
- 核心任务是否可完成

## 给 Codex 的交接提示词

```text
这是一张 UI 设计图到代码实现任务。
不要直接照图堆样式。请先完成设计解构：

1. 拆出页面结构
2. 列出组件清单
3. 把颜色、圆角、字号、阴影映射到现有 theme token
4. 标出 loading、empty、error、long text、keyboard 状态
5. 判断哪些视觉效果在 React Native / Android 上有风险
6. 给出必要降级方案

只有完成上述解构后，才开始写代码。
实现过程中优先保证真实数据、移动端可用性和状态完整，再做视觉 polish。
```
