# Feature: 作物系统模版库

## 目标

为新用户提供可导入的系统作物模版初始库，降低从零创建作物生育阶段的成本，并通过 `farm_id IS NULL` 将系统模版与用户私有模版区分。

## 非目标

- 不做系统模版后台运营管理界面。
- 不做系统模版版本升级、推送或区域化分发机制。
- 不让 Skill、LLM 或前端运行时自动写入系统模版库。
- 不把本文件中的 seed 内容视为已审核农业标准；当前内容仅为待人工审核初稿。

## 技术方案

### 涉及的模块

- models/: `CropTemplate.farm_id` 允许为空，`category` 记录作物分类。
- services/: 系统模版查询、导入和精确查重复用 `crop_service`。
- seed/: `backend/app/seed/system_crop_templates.py` 定义初始系统模版和加载函数。
- alembic/: data migration 调用 seed 模块写入 `farm_id IS NULL` 的系统模版记录。
- api/: 已提供系统模版列表与导入接口，系统模版写保护由服务层/API 层共同约束。
- admin-web/: 业务运营下提供“系统模板”平台资产入口；管理员个人 app 账号下的作物模板、种植周期、农事日志、成本记账、天气预报归入“业务调试”。

### 数据模型变更

```sql
-- 系统模版以 farm_id IS NULL 表示；用户私有模版仍为具体 farm_id。
ALTER TABLE crop_templates MODIFY farm_id INTEGER NULL;
ALTER TABLE crop_templates ADD COLUMN category VARCHAR(50) NULL;
CREATE INDEX ix_crop_templates_farm_name_variety
    ON crop_templates (farm_id, name, variety);
```

### API 变更

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | /crops/templates/system | 列出系统作物模版，支持按 category 筛选 |
| POST | /crops/templates/system/{id}/import | 将系统模版深拷贝到当前 farm |
| POST | /crops/templates | 创建用户模版前做精确查重，重复时返回已有 ID |

请求/响应体：

```json
{
  "system_template": {
    "id": 1,
    "name": "西瓜",
    "variety": "8424通用型",
    "category": "水果",
    "stages": []
  },
  "import_response": {
    "id": 101,
    "already_exists": false
  }
}
```

### Admin Web 信息架构

| 分组 | 页面 | 语义 |
|------|------|------|
| 业务运营 | 系统模板 | 管理平台预置的系统作物模板资产，模板以 `farm_id IS NULL` 标识 |
| 业务运营 | 用户管理 | 管理真实用户与账号状态 |
| 业务调试 | 业务调试中心 | 集中调试智能填写、种植与作业、用工、账务等业务 API |
| 业务调试 | 作物模板 / 种植周期 / 农事日志 / 成本记账 / 天气预报 / AI 助手 | 展示管理员个人 app 账号的数据，用于创建样例数据和排查业务上下文 |

管理员账号同时拥有 admin 权限和 app 业务数据，因此业务数据视图不再放在“业务运营”分组中，避免把个人调试数据误认为平台运营资产。

## 审核状态

- 当前 seed 数据为“待人工审核初稿”，用于工程联调和产品/农业顾问评审。
- 审核人字段统一记录为“待产品/农业顾问审核”，代理不得代签。
- 通过审核前，产品文案与运营入口不得声称这些模版为“成熟”“权威”或“已审核”。

## 验收标准

- [x] 系统 seed 模块定义 10 个常见作物模版，且每条包含分类、品种和生育阶段。
- [x] seed 加载函数重复执行不产生重复数据。
- [x] 删除某条 seed 系统模版后，再次加载可补回缺失模板。
- [ ] 产品/农业顾问完成生育阶段内容审核并签字。
