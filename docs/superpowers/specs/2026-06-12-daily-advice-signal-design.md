# 今日建议信源与优先级设计

## 背景

当前 `GET /agent/daily` 通过 `farm_context_service.build_summary()` 汇总农场现状，再交给 LLM 生成结构化 `DailyAdviceResponse`。摘要混合了活跃茬口、近期农事、欠账、本月花费和天气。这个结构能快速生成建议，但会把经营状态误当成今日行动，例如没有明确结算日期的未结人工每天都出现在「AI 今日建议」里，造成重复打扰。

新的目标是把首页信息分层：`AI 今日建议` 只展示未来 3 天最值得行动的事项；资金、成本、未结人工等经营状态继续放在资金概览或经营态势卡片中。

## 产品定位

首页信息分为三类：

```text
AI 今日建议
  = 未来 3 天最该行动的 Top 3
  = 农事、天气、到期风险优先
  = 不展示普通经营状态

资金概览
  = 未结人工、欠款、本月花费、收入支出
  = 不默认生成催收或付款建议

风险预警
  = 高温、暴雨、大风、霜冻、逾期作业、逾期账款
  = 可与 AI 今日建议共享最高优先事项
```

`AI 今日建议` 的核心问题不是“系统知道哪些状态”，而是“今天或未来 3 天用户最该处理什么”。

## 信源边界

| 信源 | 是否进入 AI 今日建议 | 用途 | 进入条件 |
| --- | --- | --- | --- |
| 天气 | 是 | 识别高温、暴雨、大风、霜冻、连续降雨等风险 | 今天或未来 3 天影响农事安排 |
| 活跃茬口 | 是 | 判断生长阶段、预计采收或关键管理窗口 | 处于关键阶段或需要结合天气/作业给出行动 |
| 作业单 | 是 | 提醒今日、临近或逾期任务 | 今日、未来 3 天、已逾期 |
| 近期农事记录 | 间接使用 | 避免重复建议刚完成的操作 | 用于去重和抑制，不直接生成建议 |
| 赊账欠款 | 条件进入 | 提醒明确到期风险 | 必须有 `due_date`，且未来 3 天到期或已逾期 |
| 未结人工 | 默认不进入 | 展示资金状态 | 只放资金概览；未来若有结算日字段，再按到期规则进入 |
| 本月花费 | 默认不进入 | 展示经营态势 | 只放经营态势或成本分析，不作为今日建议 |
| 用户设置 | 是 | 地点、提醒偏好、作物偏好 | 用于信源选择和排序权重 |
| 历史建议/反馈 | 是 | 去重、抑制、降级 | 已处理、不再提醒、连续重复时生效 |

## 优先级模型

后端先把信源转为候选事项，再排序和降噪。LLM 只负责把候选事项改写成自然、简洁的 `preview` 和 `items`，不负责决定所有业务优先级。

### P1 紧急

今天或明天不处理会带来明显损失。

- 极端天气预警。
- 高温、暴雨、大风、霜冻等会影响当前作物或作业的天气风险。
- 今日到期或已逾期作业单。
- 已逾期账款，且有明确 `due_date`。

### P2 重要

未来 3 天建议处理，会影响长势、作业节奏或经营风险。

- 未来 3 天临近的作业单。
- 当前茬口进入关键管理阶段。
- 普通但相关的天气风险，例如连续降雨影响采收或打药。
- 未来 3 天到期的账款，且有明确 `due_date`。

### P3 提醒

有帮助但不急，只在 P1/P2 不足时补位。

- 无活跃茬口时提醒创建种植周期。
- 长时间没有农事记录时提醒巡田或补记录。
- 低风险天气变化。

## 降噪规则

- 首页 `AI 今日建议` 默认只展示 Top 3。
- 详情页最多展示 Top 5。
- P1 永远优先；多个 P1 同时存在时按天气风险、逾期作业、逾期账款、关键农事窗口排序。
- 同一类别首页限制数量：
  - 天气最多 1 条。
  - 资金最多 1 条，且必须有明确到期日。
  - 农事最多 2 条。
- 无 `due_date` 的欠款不进入 AI 今日建议。
- 未结人工当前没有结算日期字段，不进入 AI 今日建议。
- 本月花费不进入 AI 今日建议。
- 近 3 天已完成的同类农事不重复建议。
- 连续多天同类天气合并为一条，例如“未来 3 天持续高温，避开中午采收”。
- 后续支持用户反馈后，已处理或不再提醒的事项应在一段时间内抑制。

## 架构设计

当前 `farm_context_service.build_summary()` 是单段文本摘要，后续应拆成候选生成和摘要渲染两个阶段：

```text
数据库 / 天气服务
  ↓
DailySignalCollector
  收集天气、茬口、作业单、近期农事、到期账款、用户设置、历史建议
  ↓
DailyAdviceCandidateBuilder
  把信源转为结构化候选事项
  ↓
DailyAdviceRanker
  计算 priority、时间窗口、影响范围和新鲜度，执行降噪
  ↓
DailyAdvicePromptRenderer
  只把入选候选交给 LLM 改写
  ↓
DailyAdviceResponse
  preview + items + created_at
```

建议保留 `GET /agent/daily` 入口和 `DailyAdviceResponse` 输出结构，不把今日建议整体封装为 Skill。底层信源可以继续复用已有服务或 Skill 的查询逻辑，但今日建议的排序和降噪应由 application/service 层控制。

## 候选事项结构

候选事项应使用结构化对象，避免在排序前丢失业务含义：

```text
DailyAdviceCandidate
  id: 稳定去重键
  category: weather | operation | crop_stage | finance | setup | record
  title_hint: 候选标题
  detail_hint: 候选细节
  priority: 1 | 2 | 3
  due_date: 可选日期
  source_type: weather | work_order | crop_cycle | cost_record | system
  source_id: 可选来源 ID
  dedupe_key: 同类建议合并键
  reason: 排序原因，供 trace/debug 使用
```

## 缓存策略

当前实际实现使用 `AgentRecord(record_type="daily")` 做当日缓存。后续缓存键应至少考虑：

- `farm_id`
- 日期
- 活跃茬口集合
- 未来 3 天天气风险摘要
- 今日/未来 3 天作业单版本
- 到期账款集合

刷新接口 `POST /agent/daily/refresh` 应绕过缓存，重新收集信源和排序。

## 可观测性

为了后续调试“为什么今天推了这条”，建议在 trace 或 debug export 中记录：

- 收集到的候选数量。
- 入选 Top 3/Top 5。
- 被过滤的候选及原因，例如 `no_due_date`、`recently_done`、`category_limit`、`duplicate_weather`。
- LLM 改写前的结构化候选。

## 测试策略

后端应覆盖以下场景：

- 有高温预警时，天气候选进入 P1，并排在普通建议前。
- 未结人工存在但无结算日期时，不进入 AI 今日建议。
- 赊账无 `due_date` 时不进入 AI 今日建议。
- 赊账有 `due_date` 且 3 天内到期时进入 P2，已逾期时进入 P1。
- 近期 3 天已完成的同类农事不重复建议。
- 同类天气连续多天时合并为一条。
- 首页最多返回 3 条，详情最多返回 5 条。

## 非目标

- 不重做首页 UI。
- 不新增未结人工结算日期字段。
- 不把今日建议整体改造成 Skill。
- 不在本设计中实现用户反馈抑制表，只预留规则边界。
