## 3. 标准请求生命周期

以「农户说：上个月买了 200 块化肥，老王农资店赊账」为例：

```
1. Flutter App 输入 → POST /agent/chat/stream
2. bootstrap middleware：JWT 校验 → 解析 farm_id → 注入 trace_id
3. domains/conversation/routes.py 转发到 application/chat/use_case.py 或 stream_chat.py
4. application/chat：
   ├─ 调用 conversation_service 保存用户消息
   ├─ 调用 pending_actions 检查是否有待确认动作
   └─ 准备 AgentState 并调用 runtime loop / stream
5. agent/guardrails + agent/router：
   ├─ check_input（Guardrails）
   ├─ classify_intent（问候/记账/查询/闲聊）
   └─ 生成 RouterDecision 与候选工具
6. agent/runtime：
   ├─ nodes._llm_node 准备 LLM 调用
   ├─ router.service + tool_selector 给出候选 Skill
   ├─ context.builder 构造 ContextBundle（farm/cycle/ledger/memory）
   ├─ prompt.composer 渲染 system prompt + skill 摘要
   └─ 调用 LLM 拿到 tool_calls
7. runtime.tool_executor：
   ├─ 解析 tool_calls（如 create_cost_record）
   ├─ 校验权限（write_confirm?）
   ├─ 若是写操作 → 入 pending_actions，返回确认话术
   └─ Reflector 检查 tool_call 与回复一致性
8. response.formatter：
   ├─ 普通回复：直接 JSON
   └─ 流式回复：SSE 事件（token / tool_call / pending / done）
9. application 提交 Memory observation event
10. observability 提交 trace + token_stats
11. conversation_service 保存 assistant 消息
12. Flutter App 渲染：「已记账：化肥 200 元，向老王农资店赊账，确认？」
13. 用户确认 → pending plan 执行器真正写入
```

## 5. 关键设计决策

### 5.1 为什么 Agent 平台是核心，不是业务模块？

业务模块（crop/cycle/ledger）逻辑稳定且简单，**真正难的是让 Agent 正确理解农户意图、选择正确 Skill、构造合适上下文、生成可信回复**。Agent 平台的工程化深度直接决定产品上限。

### 5.2 为什么当前用自研 ReAct loop，而不是 LangChain Agent？

- 多步骤任务需要显式状态流转（记账 → 确认 → 写入 → 反馈）
- 流式回复需要 SSE 事件流（token / tool_call / pending / done）
- 调试需要可控 trace + replay，而不是黑盒 agent 调度
- 当前代码在 `agent/runtime/loop.py` 使用纯 Python ReAct loop，降低框架耦合

不选 LangChain Agent 是因为它的 message 流转黑盒，无法在 2s SLA 下精准调优；直接裸调 SDK 又缺少统一的 tool execution、reflection、trace 和 pending plan 约束。当前折中是 LangChain message/tool 适配 + 自研 runtime loop。

### 5.3 为什么 Skill 必须有 `skill.md`？

- LLM 需要结构化元数据来决定何时调用（trigger / parameters / 不要使用）
- 工程需要单一契约来注册、校验、生成 API 文档
- 测试需要 fixture
- 文档与代码强制对齐，避免「实现改了文档没改」

详见 `.claude/rules/skill-writing.md`。

### 5.4 为什么 Memory 暂时不接外部 RAG？

2 核 4G 部署不内置 embedding 模型 + 向量库 + 重排。当前长期记忆已有 `memory_records` 表和 `memory/long_term` 存储，外部 QuillRAG 通过 `rag_service` 配置作为可选只读检索；未配置时降级为空知识块，Agent 靠短时记忆 + 业务工具 + Prompt 正常执行。

### 5.5 为什么写操作必须 Pending？

农业场景写错的代价高（误删账、误记账、误改茬口）。所有写 Skill 默认 `direct_call: false`，必须经过 Pending Action 二次确认。Reflector 还会检查「回复里说执行了但 tool_call 没成功」的幻觉执行。

### 5.6 为什么 DataFlywheel 是一等公民？

不是事后补丁，而是从第一天就埋的护城河。真实会话比评测集宝贵 10 倍，且免费。飞轮闭环让产品自我进化，是中小团队对抗大厂的唯一路径。

### 5.7 为什么 DataFlywheel 必须有 Discovery Layer？

农场场景下 **95%+ 的真实会话是正常会话**（"你好"、"今天天气怎么样"、"你是谁"）。让标注员从全量会话中人工捞坏例，效率极低、覆盖极差、反馈延迟 > 3 天。

Discovery Layer 通过 **规则命中 + LLM Judge** 两级信号计算风险，默认按风险倒序推送，自动隐藏低风险会话。让标注员只看值得标注的样本，这是数据飞轮从「能跑」到「能转」的关键升级。设计原则：不引入新基础设施（复用 MySQL/Mongo + 后台任务 + 现有 LLM client）、不学权重（1000+ 标注前手工权重足够）、Judge 不作为最终真值。

详见 [01.06 数据飞轮与评测 § 9 Discovery Layer](../01_正式设计/06_数据飞轮与评测.md)。

## 6. Farm Manager 架构取舍

| 维度 | 当前设计 |
| --- | --- |
| 客户端 | Flutter App + Admin Web，文本交互为主，REST/SSE 协议 |
| 任务复杂度 | 聚焦记账、茬口、工资、天气、报表等农场经营任务，允许跨茬口和多轮上下文 |
| Agent 形态 | 单主 Agent + 13 个 Skill 包，每个包可暴露多个 operation / alias |
| 响应策略 | 查询和建议保持秒级体验；写操作优先保证确认、回溯和一致性 |
| 记忆形态 | 农场历史、茬口、赊账、偏好、别名等跨周期记忆 |
| 数据飞轮 | 从 P0 起作为一等公民，服务样本治理、回归、评测和修复闭环 |
| 部署形态 | 2C4G 单机优先，MySQL + JSONL + 异步任务 |
