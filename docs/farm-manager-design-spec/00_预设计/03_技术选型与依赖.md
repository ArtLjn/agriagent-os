## 2. 关键选型决策

### 2.1 为什么不用 LangChain Agent / LangGraph 全托管？

LangChain Agent 的 message 流转黑盒，无法在 2s SLA 下精准调优。ContextBundle、Prompt Composer、Memory Service 等核心逻辑高度定制，框架会引入不必要的黑盒。

当前代码进一步收敛为 `app/agent/runtime/loop.py` 的纯 Python ReAct loop：保留 LangChain Core 的 message/tool 类型适配，但把 LLM 调用、并行工具执行、反思、Pending 和 trace 交给本项目显式控制。

### 2.2 为什么不用向量数据库？

- 2C4G 跑不动独立向量库 + embedding 模型 + 重排
- 当前记忆规模 < 1 万条，关系型 MySQL + LIKE/全文索引够用
- 预留 MemoryService 端口，未来独立 RAG 服务可平滑接入

### 2.3 为什么不用 Kafka / ClickHouse？

- 2C4G 资源约束
- 日志量不大（日 < 10GB），JSONL + MySQL 索引够用
- 异步任务用 APScheduler 或 cron 替代

### 2.4 为什么 Admin Web 用 Ant Design 而不是 Shadcn？

- 后台管理 AntD 生态更成熟（复杂表格、表单、图表开箱即用）
- 团队熟悉度更高
- 移动端 App 单独走 Flutter 设计系统，详见 [.claude/rules/app-ui.md](../../../.claude/rules/app-ui.md)

### 2.5 为什么移动端是 Flutter？

当前代码已是 Flutter（`mobile-app/lib/` + `pubspec.yaml`）。选型原因：
- RN 在复杂动画/手势体验差
- Flutter 一套代码三端（Android/iOS/Web）成本更低
- Dart 类型系统更强
- Flutter 设计上限高（Material/Cupertino 自定义容易）

### 2.6 为什么后端 Python 而不是 Go/Rust？

- AI 生态：OpenAI-compatible SDK、LangChain Core message/tool 类型、Python Agent 工程生态成熟
- 团队熟悉度
- 业务量级（200 农户）Python 性能够用
- 后期如果某个 Skill 是性能瓶颈，可以用 Go/Rust 重写为独立服务

## 3. 依赖管理

| 端 | 工具 | 锁文件 |
| --- | --- | --- |
| Backend | pip / venv | `backend/requirements.txt` |
| Admin Web | pnpm | `admin-web/package.json` + `pnpm-lock.yaml` |
| Mobile | pub | `mobile-app/pubspec.yaml` + `pubspec.lock` |

**禁止规则**：
- 不允许直接 `pip install`、`npm install -g`、`flutter pub add` 后不更新对应依赖清单
- 升级依赖必须开 PR 说明理由和兼容性
- 安全扫描：后端依赖清单 + 前端 `pnpm audit`，每月一次

## 4. 外部服务依赖

| 服务 | 用途 | 配置位置 | 失败兜底 |
| --- | --- | --- | --- |
| DashScope/OpenAI-compatible LLM | 默认 LLM | `shared/llm_client_manager.py` + `config.yaml` | Provider fallback / 降级提示 |
| 其他 OpenAI-compatible Provider | 可选兜底 LLM | `providers.json` + `config.yaml` | 限流降级 + 错误提示 |
| QWeather | 主天气 | `domains/weather/providers/qweather.py` | 切换 Open-Meteo |
| Open-Meteo | 兜底天气 | 同上 | 提示「天气服务暂时不可用」 |
| LangSmith（可选） | Trace 可视化 | `shared/config.py` | 不影响主流程 |
| MySQL | 业务数据 | `shared/database.py` | 主备切换（生产） |
| MongoDB（可选） | trace / 会话 / 飞轮文档存储 | `infra/mongo.py` + `infra/repository_runtime.py` | 可按 storage 配置回退 MySQL |
| 文件系统 | 修复包、导出文件、部分 event log | `data/` / `logs/` | 不可缺失 |

## 5. 部署拓扑

### 5.1 生产（腾讯云生产服务器）

```
Internet
   │
   ▼
Nginx (443 TLS + 80 redirect)
   │
   ├── / → admin-web 静态资源
   ├── /agent、/auth、/smart-fill、/planting 等 → uvicorn :8000
   └── /agent/chat/stream → uvicorn (SSE 长连接)
   │
   ▼
uvicorn (FastAPI, Python 3.12)
   │
   ├── MySQL 8.x（本地或远端）
   ├── LLM Provider（公网）
   ├── QWeather / Open-Meteo（公网）
   └── JSONL 文件（本地 <repo_dir>/backend/logs/）

systemd: <service_name>.service（具体名称见部署仓库）
日常运维：见部署仓库 deploy/ 目录
```

### 5.2 开发

```bash
# Backend
cd backend && python -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload

# Admin Web
cd admin-web && pnpm install && pnpm dev

# Mobile
cd mobile-app && flutter pub get && flutter run
```

## 6. 版本治理

| 维度 | 当前版本 | 升级策略 |
| --- | --- | --- |
| Python | 3.12 | 跟随 Ubuntu LTS，不主动升 |
| FastAPI | 最新稳定 | 半年评估一次 |
| LangChain Core / OpenAI adapter | requirements 固定版本 | 模型或 tool-call 协议升级时评估 |
| React | 19 | 跟随官方 |
| Ant Design | 5.x | 跟随官方 |
| Flutter | 3.x stable | 跟随官方 |
| MySQL | 8.x | LTS |

**模型版本**：当前默认 `qwen3.6-35b-a3b`（DashScope OpenAI-compatible）。升级模型必须：
1. 在 Evaluation 跑核心 regression suite
2. 对比上一版本通过率、token、延迟
3. 文档记录在 [04_相关规范/01_Prompt与人设规范](../04_相关规范/01_Prompt与人设规范.md)
