# 07 — 可观测与运维

> 状态：草稿 | 维护：BlockShip | 关联：[01_Agent平台架构](./01_Agent平台架构.md)、[06_数据飞轮与评测](./06_数据飞轮与评测.md)

---

## 1. 为什么

2C4G 单机部署，但**不能因为小就忽略可观测**。每次写操作、每次工具调用、每条 LLM 请求都要能追溯：

- 用户说「上次给我记错了」→ 我能找到那条 trace
- 数据飞轮标注 bad case → 我能复制完整 debug JSON
- LLM 调用超时 → 我能看到是 Provider 问题还是 prompt 太长
- Token 成本飙升 → 我能定位到哪个 Skill 哪个用户

## 2. Trace 模型

每轮对话的 trace 结构：

```python
class TraceRecord:
    trace_id: str            # 全局唯一
    session_id: str          # 会话 ID
    turn_id: str             # 本轮 ID
    request_id: str          # HTTP 请求 ID
    farm_id: int             # 农场
    user_id: int             # 用户
    started_at: datetime
    ended_at: datetime

    # 链路段
    spans: list[Span]

class Span:
    span_id: str
    parent_id: str | None
    node: str                # entry / guardrail / router_plan / context_build / prompt_compose / llm_call / tool_call / reflect / response
    started_at: datetime
    ended_at: datetime
    duration_ms: int

    input_summary: dict      # 脱敏后摘要
    output_summary: dict
    error: dict | None

    # 关联段
    llm_meta: LLMMeta | None       # model / token / cost / cached
    tool_meta: ToolMeta | None     # skill_name / status / params_summary
    prompt_meta: PromptMeta | None # version / snippet_versions / token_before_budget
```

## 3. 存储分层

| 层 | 存储 | 用途 |
| --- | --- | --- |
| **热索引** | MySQL `trace_records` 表 | 列表查询：farm_id + started_at |
| **Mongo 节点明细** | Mongo `traceRecords` 集合 | 一条执行节点一个文档，保留 input/output/token/error 明细 |
| **Mongo 请求摘要** | Mongo `traceRequests` 集合 | 一轮 request 一个摘要文档，供 TraceMonitor 列表和根因卡片快速读取 |
| **明细** | JSONL 文件 `logs/traces/{date}/{trace_id}.jsonl` | 详情页：按 trace_id 直接读 |
| **聚合** | MySQL `token_stats` / `skill_call_stats` | Dashboard：按天聚合 |

**禁止**：
- 热索引存全量 span（一行 trace 一行 MySQL）
- JSONL 列表扫描（必须先查 MySQL 索引拿文件路径）
- 同步阻塞主流程写 JSONL（必须 fire-and-forget 或 async）

Mongo 写入策略：
- `traceRecords` 保持节点级 upsert，唯一键为 `mysqlId`，兼容 MySQL 双写和 Mongo-only。
- `traceRequests` 保存 request 级摘要，唯一键为 `farmId + requestId`，字段包含 `status`、`statusReason`、`rootError`、`metrics`、`nodeCount`、`totalDurationMs`、`startedAt`、`endedAt`。
- TraceMonitor 请求列表优先读取 `traceRequests`；老数据没有摘要时回退聚合 `traceRecords`，确保历史 trace 可看。
- `traceRequests` 与 `traceRecords` 使用同样 TTL，避免节点清理后摘要残留。
- `traceRecords` 是回放和节点详情来源，`traceRequests` 只服务默认展示、列表分页、告警和根因入口。
- TraceDAO 禁止把结构化 JSON 直接截成字符串。默认保留 128KB 内的完整 JSON；超过上限时只裁剪长字符串和长数组叶子，并写入 `__trace_truncated` 与 `__trace_original_json_len`。

TraceMonitor 默认展示策略：
- 列表卡片显示 `status`、`statusReason`、节点数、耗时、时间和 request/session。
- 展开后先展示 `Trace 摘要`、`根因`、`执行指标`，再展示 Gantt timeline。
- 根因卡片展示 `rootError.code`、`rootError.message`、`rootError.recover` 和对应节点。
- 节点抽屉补充 `error_code` 和 `recover`，同时保留 Context/RAG/Skill 原始详情展开。

Playground LLM Context 调试策略：
- `context_build/context_bundle` 展示上下文块选择、预算、压缩和丢弃结果；它不是最终模型输入。
- `prompt_budget/final_prompt` 展示最终 prompt token 预算摘要；它也不是最终模型输入。
- `prompt_budget/final_llm_context` 展示预算处理后真正送入 LLM 的快照，包含完整 `system_prompt`、压缩后的 `messages[]`、`context_blocks[]` 和 `budget`。
- Playground 默认收起 `LLM 上下文快照` 面板，只在当前 request timeline 存在 `final_llm_context` 节点时展示；展开后用于排查上下文压缩、历史摘要、工具结果压缩和“失忆”问题。
- `final_llm_context.messages[]` 必须保留 `role`、`type`、`content`、`tool_calls`、`invalid_tool_calls`、`tool_call_id`、`name` 等 LLM 输入相关字段。
- `final_llm_context` 复用 TraceDAO 结构化裁剪策略：128KB 内完整保留；超过上限时只裁剪长字符串和长数组叶子，并写入 `__trace_truncated` 与 `__trace_original_json_len`。

## 4. 关键观测点

| 节点 | 观测内容 | 告警阈值 |
| --- | --- | --- |
| **request_entry** | 完整请求、用户、farm、session | — |
| **guardrail** | check_input / filter_output 命中规则 | 命中率 > 5% warning |
| **router_plan** | intent、candidates、tool_selector / PlanDraft 决策 | 候选为空率 > 10% warning |
| **context_build** | selector 命中、token budget、压缩触发 | 超 budget 频率 > 20% warning |
| **prompt_compose** | snippet_versions、最终 token | — |
| **llm_call** | model、input/output tokens、latency、cached、cost | latency p95 > 5s critical；失败率 > 5% critical |
| **tool_call** | skill_name、status、duration、params_summary | 失败率 > 5% warning |
| **reflection** | checks 触发、reflection_trace 写入 | 触发率 > 10% warning |
| **pending_action** | 创建、确认、取消、过期 | 过期率 > 30% warning |
| **response_output** | 完整响应、SSE 事件数 | — |
| **memory_observe** | observation event 写入 | 失败 retry ≥ 3 critical |

### 5.1 日志输出优化

日志不是 trace 明细的替代品。生产日志用于快速 grep、告警和定位 `request_id`；Trace 用于展开单轮执行链路。日志每行必须是单行、短文本、结构化字段稳定，避免把完整 prompt、完整 tool args、完整 provider error 和多行工具结果直接写入 `app.log`。

当前日志的主要问题：
- 同一类事件缺少固定 `event` / `code` / `step_id`，只能靠中文 message 解析。
- 部分日志把关键字段拼进 message，机器统计时不稳定。
- 工具结果可能多行输出，导致一条事件在 `app.log` 中跨行，破坏 grep 和日志采集。
- `uvicorn.access` 没有关联 `request_id`，只能靠时间邻近推断。
- LLM provider 原始错误可能携带敏感信息或过长 URL，应输出脱敏后的 `error_code` 和短 message。
- trace 已经记录的详情不应在日志重复输出，避免日志膨胀和隐私风险。

日志分为两类：

| 类型 | 文件 | 用途 | 内容 |
| --- | --- | --- | --- |
| 人读运行日志 | `logs/app.log` | 本地排障、快速 grep | 固定前缀 + `event=... code=... key=value` |
| 机器结构化日志 | `logs/app.jsonl` | 后续接入 Loki / ELK / Vector | 单行 JSON，字段稳定 |

`app.log` 推荐格式：

```text
2026-07-24 14:54:22,991 │ 74248e50 │ app.agent.runtime.tool_executor │ INFO │ event=llm_tool_calls_selected step_id=llm-generation-2 status=success tool_calls=2 tools=manage_workers,manage_workers model=fallback-generation-model
2026-07-24 14:54:23,340 │ 74248e50 │ app.agent.runtime.tool_pending │ WARNING │ event=pending_plan_contract_blocked code=pending_plan_contract_blocked step_id=pending-plan-contract-1 source=plan_draft tool=manage_workers missing_fields=name diagnosis=pending_step_missing_field_present_in_llm_tool_call detail_ref=trace://trace_20260724_74248e50/steps/pending-plan-contract-1
2026-07-24 14:54:31,375 │ 74248e50 │ app.application.chat.stream_tail │ INFO │ event=agent_stream_completed status=blocked duration_ms=12384 reply_len=52 skills=pending_plan root_error_code=pending_plan_contract_blocked trace_id=trace_20260724_74248e50
```

日志字段规范：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `timestamp` | 是 | ISO 时间，带时区 |
| `level` | 是 | `INFO` / `WARNING` / `ERROR` |
| `logger` | 是 | Python logger 名称 |
| `event` | 是 | 稳定事件名，禁止写自由中文句子 |
| `code` | 异常必填 | 稳定错误码或诊断码 |
| `trace_id` | 有 trace 时必填 | 一轮 trace 全局 ID |
| `request_id` | 是 | HTTP 请求 ID |
| `session_id` | Agent 请求必填 | 会话 ID |
| `turn_id` | Agent 请求必填 | 本轮 ID |
| `step_id` | step 事件必填 | 对应 trace step |
| `status` | 是 | `started` / `success` / `failed` / `blocked` / `timeout` |
| `duration_ms` | 结束事件必填 | 当前事件耗时 |
| `labels` | 否 | 低基数字段，用于聚合 |
| `data` | 否 | 小型摘要，禁止大 prompt / 大响应 |
| `error` | 异常必填 | 脱敏后的错误类型、消息和恢复建议 |

日志与 trace 的边界：
- 日志只放摘要、错误码、耗时、数量、短 preview 和 `detail_ref`。
- Trace step 放排障所需的结构化输入输出摘要。
- Debug Bundle 才放完整 prompt、history、工具原始入参、RAG chunk 和 replay 所需上下文。
- 同一个事件必须同时能通过 `request_id` grep 日志，通过 `step_id` 跳到 trace 节点。

落地顺序：
1. 先统一事件命名和错误码，不改变存储。
2. 再给 `app.log` 增加稳定 `event=... code=... step_id=... status=...` 字段。
3. 然后增加可选 `app.jsonl` handler，供后续接入日志系统。
4. 最后把 `uvicorn.access` 与 request context 打通，确保访问日志也能关联 `request_id`。

## 6. 监控指标（Metrics）

### 6.1 请求级

| 指标 | 类型 | 含义 |
| --- | --- | --- |
| `agent_request_total{status}` | counter | 每天调用量，status: success/fail/timeout |
| `agent_request_latency_seconds{quantile}` | histogram | 延迟分位 |
| `agent_request_concurrent` | gauge | 当前并发数 |

### 6.2 LLM 级

| 指标 | 类型 | 含义 |
| --- | --- | --- |
| `llm_call_total{model, cached}` | counter | 大模型调用量 |
| `llm_latency_seconds{model}` | histogram | 大模型延迟 |

## 9. 日志规范

```python
logger = logging.getLogger(__name__)

logger.info(
    "agent_request_started",
    extra={
        "trace_id": trace_id,
        "session_id": session_id,
        "farm_id": farm_id,
        "intent": intent.value,
        "candidates": candidates,
    },
)
```

**禁止**：
- `print(...)` 调试
- `logger.debug` 写入生产（生产只 info 及以上）
- 日志带未脱敏的用户密码、token、个人信息
- 同步写入阻塞主流程

日志带 `trace_id` 是硬约束，CI 用 `scripts/check-log-format.sh` 验证。
