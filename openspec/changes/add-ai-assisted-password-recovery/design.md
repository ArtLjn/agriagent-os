## Context

项目当前已有手机号 + 密码登录、JWT 鉴权、管理员用户管理能力。用户场景是小范围熟人农场 App，个人开发者暂不适合接入短信验证码、邮箱找回或 OAuth。密码已哈希存储，系统不能也不应该取回明文密码，因此找回密码应转化为“重设密码”。

这次设计的目标是让用户体验比“直接找管理员”更顺，同时不把敏感操作交给 LLM：找回入口默认是“AI 外观的流程助手”，用固定话术沟通和收集，管理员负责确认和发码，用户用一次性重置码自己设置新密码。只有用户输入特别口语化、字段难以抽取时，才可选调用一次轻量 LLM 做结构化解析。

## Goals / Non-Goals

**Goals:**
- 提供不依赖短信、邮箱、OAuth 的密码恢复流程。
- 登录页支持 AI 客服找回入口和重置码改密入口。
- AI 客服/流程助手只能创建找回申请，不直接改密码、不泄露账号信息，默认不消耗 LLM token。
- 管理员可以查看找回申请、匹配用户、生成一次性重置码。
- 用户可以使用账号/手机号 + 重置码 + 新密码自行重设密码。
- 所有重置码只保存哈希，有有效期，只能使用一次，并记录管理员操作。

**Non-Goals:**
- 不恢复或展示原密码。
- 不自动验证真实身份，不引入实名、短信、邮箱或第三方 OAuth。
- 不让 AI 自动执行重置密码、改绑手机号或生成重置码。
- 不把登录页找回入口接入通用农事 Agent。
- 不做复杂客服工单系统，只做轻量找回申请状态流转。

## Decisions

### 1. AI 客服只创建找回申请

**选择**: AI 客服通过受限工具创建 `password_recovery_request`，收集用户输入的手机号/账号、昵称、备注或农场线索。它不得调用重置密码、改绑手机号、生成重置码等工具。

**理由**: LLM 适合解释流程和收集信息，但不适合作为身份判定主体。敏感操作必须由管理员确认。

**替代方案**: 让 AI 根据农场历史信息自动判断并发码。该方案体验更自动，但错误恢复账号的风险不可接受。

### 2. 找回入口默认使用流程助手，不默认调用 LLM

**选择**: 登录页“问问小助手”默认实现为表单式/脚本式流程助手：固定 2-3 步收集账号线索和备注，创建找回申请。范围外问题直接固定回复，不调用 LLM。只有当用户一段话里混合了手机号、昵称、备注且需要抽取字段时，才可选调用一次轻量 LLM。

**理由**: 未登录入口无法按 user_id 控制 token 配额，接入通用 Agent 容易被反复骚扰导致 token 浪费。流程助手能保留“像客服”的体验，同时成本为零且行为可控。

**替代方案**: 直接接入现有 Agent Chat。实现复用度高，但它会暴露大模型对话面，成本和越权风险都更高。

### 3. 管理员生成一次性重置码，用户自行改密

**选择**: 管理员在后台处理申请时选择目标用户并生成重置码。系统只展示一次明文重置码，数据库保存哈希。用户在 App 输入账号/手机号、重置码和新密码完成重设。

**理由**: 管理员不需要知道用户新密码，用户也不用依赖短信或邮箱。一次性码比管理员直接设置固定临时密码更安全。

**替代方案**: 管理员直接重置为临时密码。实现更简单，但临时密码会被管理员看到，且用户可能长期不修改。

### 4. 重置码使用短码但保持足够随机性

**选择**: 重置码格式使用易读短码，例如 `FM-4829-QL7K`。随机部分至少 8-10 位，排除易混字符。有效期默认 24 小时；小范围使用可配置到 7 天。使用后立即失效。

**理由**: 用户可能年龄较大，过长 token 输入体验差。短码需要配合绑定用户、有效期、一次性和限流来降低风险。

**替代方案**: 使用长 URL token。安全性更高，但用户没有邮箱/短信渠道时传递和输入都不方便。

### 5. 账号枚举风险用模糊响应处理

**选择**: 找回申请提交和重置码校验失败时，不向匿名用户明确暴露“账号是否存在”。AI 客服和 API 响应使用“已收到申请/信息不匹配或已过期”等模糊文案。

**理由**: 即使是小范围 App，也不应该让外部用户批量探测手机号是否注册。

**替代方案**: 为了方便用户直接提示账号不存在。体验略好，但会增加枚举风险。

### 6. 管理后台处理轻量状态机

**选择**: 找回申请状态为 `pending`、`matched`、`resolved`、`rejected`。管理员可以添加备注、关联用户、生成重置码、标记已处理或拒绝。

**理由**: 状态足够覆盖小范围运营需要，避免过度设计工单系统。

**替代方案**: 不保存申请，只让用户直接联系管理员。实现最简单，但无法把 AI 客服收集的信息交给管理员，也缺少记录。

### 7. 未登录找回入口单独限流

**选择**: 找回助手按 IP、设备指纹或本地访客 ID、账号线索进行限流。单个找回会话最多 3-5 轮，超过后提示“已收集足够信息，请等待管理员处理”。同手机号/账号线索每天最多创建 1 条有效申请，同设备每天最多创建少量申请。

**理由**: 找回入口是公开入口，不能依赖登录后用户配额；轮次和申请限制能防止恶意消耗资源。

**替代方案**: 只做后端接口限流，不限制对话轮次。能防刷申请，但如果接入 LLM 仍可能浪费 token。

## Risks / Trade-offs

- [Risk] 重置码被别人看到后尝试使用。→ Mitigation: 绑定用户、有效期、一次性、失败限流，必要时管理员重新作废并生成新码。
- [Risk] AI 客服错误承诺“已帮你重置”。→ Mitigation: 系统提示词和工具 schema 明确 AI 只能提交申请，回复模板固定为“等待管理员处理”。
- [Risk] 未登录用户反复骚扰 AI 浪费 token。→ Mitigation: 默认流程助手不调用 LLM；范围外问题固定回复；可选 LLM 解析最多每会话 1 次，并叠加 IP/设备/账号线索限流。
- [Risk] 管理员误匹配用户。→ Mitigation: 后台展示候选用户和农场摘要，不自动选择；所有操作记录管理员和备注。
- [Risk] 匿名接口被刷。→ Mitigation: 找回申请创建和重置码校验接口加 IP/账号维度限流。
- [Risk] 小范围场景实现仍偏多。→ Mitigation: 第一阶段只做申请、发码、重设密码，不做改绑手机号和复杂工单通知。

## Migration Plan

1. 新增找回申请和重置码数据模型及迁移。
2. 新增 Auth 公开接口：创建找回申请、使用重置码重设密码。
3. 新增管理员接口：查看申请、关联用户、生成重置码、处理申请。
4. 新增登录页流程助手和受限工具，只能创建找回申请；可选轻量 LLM 解析必须受轮次和调用次数限制。
5. App 登录页增加 AI 客服入口和重置码改密页面。
6. admin-web 增加找回申请处理入口。
7. 补充限流、测试和操作日志。

Rollback 策略：可关闭登录页 AI 客服入口和重置码改密入口；已有未使用重置码可标记失效，不影响原手机号密码登录。
