## Context

当前模型中 `crop_cycles` 同时承载“种植周期”和“地块名”，`farm_logs` 强制绑定单个 `cycle_id`，`cost_records` 只能通过分类和备注表达人工费。这能支撑早期记事本式 MVP，但在春茬西瓜二三十亩、多棚、多工人的场景里会造成重复创建茬口、重复记录日志和人工账单。

本设计将 MVP 收敛为“批次 + 单元 + 作业单 + 轻量用工”。批次表达一批作物的生产周期，单元表达棚/地块/区域，作业单表达真实农事动作，用工记录表达人工工资明细。账单仍作为财务汇总入口，不升级成复杂财务系统。

## Goals / Non-Goals

**Goals:**

- 通用作物可按种植批次管理，不要求每亩或每棚创建独立茬口。
- 一个种植批次可维护多个种植单元，支持二三十亩、多棚、多地块的轻量表达。
- 农事作业单支持作用到整个批次或部分种植单元。
- 西瓜内置高频作业类型，降低农户录入成本。
- 作业单可记录参与工人和工资，人工成本自动进入成本统计。
- 保持旧茬口、日志、账单 API 可渐进兼容，降低迁移风险。

**Non-Goals:**

- 不做 GIS 地块边界、地图测亩或遥感能力。
- 不做完整 HR、排班、考勤审批、工资条、社保等人事系统。
- 不做双式记账、账户余额、银行流水或复杂财务凭证。
- 不做所有作物的专家级模板库；MVP 只内置西瓜高频作业，其他作物允许自定义。
- 不强制一次性迁移全部历史日志和账单。

## Decisions

### 继续使用 CropCycle 作为种植批次

`crop_cycles` 的产品语义升级为“种植批次/茬口”，新增总面积、季节、批次备注等字段；旧 `field_name` 保留为兼容展示字段，不再作为唯一地块表达。

替代方案是新增 `planting_batches` 并废弃 `crop_cycles`。该方案语义更干净，但会一次性影响成本、日志、Agent、报告和移动端大量接口，不适合 MVP。

### 新增 PlantingUnit 表表达棚/地块/区域

新增 `planting_units`，字段保持轻量：`farm_id`、`cycle_id`、`name`、`area_mu`、`planted_date`、`status`、`note`。MVP 不引入坐标、多边形、土壤指标等复杂字段。

替代方案是把多个棚名和面积存成 JSON。该方案改动小，但无法可靠筛选、关联作业单和统计面积。

### 新增 OperationWorkOrder 作为农事作业单

新增 `operation_work_orders`，记录作业类型、日期、作用范围、备注、照片、关联批次和可选来源。作业范围通过 `scope_type` 表达：`cycle`、`unit`、`farm`。当作用到多个种植单元时，用关联表 `operation_work_order_units` 保存。

替代方案是继续扩展 `farm_logs`。该方案短期字段少，但旧日志模型强依赖单个 `cycle_id`，难以表达“东大棚 1-6 号一起授粉”。

### 轻量 Worker 和 LaborEntry，不做完整工资系统

新增 `workers` 保存工人档案，新增 `labor_entries` 挂到作业单，记录工人、计薪方式、数量、单价、应付、已付、未付和结算状态。一个作业单可同时录入多个工人。

人工费汇总到 `cost_records`：MVP 可在作业单提交时生成一条分类为“人工”的成本记录，并通过来源字段或扩展字段关联作业单；如果暂不改 `cost_records` 来源字段，则先使用 `note` 和 `counterparty` 兼容展示，但服务层必须保留可追溯关系。

替代方案是每个工人每天直接生成一条账单。该方案能统计成本，但作业上下文会散落在备注里，后续无法按作业和地块复盘。

### 西瓜模板是快捷作业类型，不是硬编码流程引擎

西瓜 MVP 内置作业类型包括：定植、补苗、整枝打杈、理蔓、压蔓、人工授粉、留瓜/疏瓜、垫瓜/翻瓜、浇水、冲肥、打药、采收、装车。模板只提供快捷选择、默认字段提示和排序，不强制所有西瓜批次必须按固定流程走。

替代方案是做完整作物 SOP 流程和提醒引擎。该方案更完整，但会提前引入大量农业知识维护成本。

## Risks / Trade-offs

- 旧 `FarmLog` 与新作业单并存会造成列表来源复杂 → MVP 统一在服务层提供“近期农事视图”，合并旧日志和新作业单。
- 自动生成人工成本可能和手工账单重复 → 作业单生成的人工成本需要标记来源，并在 UI 上提示“来自作业单”。
- 工人同名导致统计混乱 → 工人档案使用独立 ID，录入时允许显示姓名但实际关联 ID。
- 西瓜模板过细会压迫通用作物体验 → 内置模板仅在作物为西瓜时优先展示，其他作物默认显示通用作业类型和自定义入口。
- 迁移范围过大影响当前迭代 → 先新增能力和新页面入口，旧日志和旧账单保持可读，历史数据只做兼容展示。

## Migration Plan

1. 新增表和可空字段，不删除旧字段。
2. 为现有 `crop_cycles.field_name` 生成可选默认种植单元；无法解析的保留为批次备注。
3. 新作业单写入新表；旧 `farm_logs` 继续可读。
4. 作业单人工费生成或关联 `cost_records`，账单列表显示来源。
5. 农场上下文、报表、Agent 查询优先读取新作业单，并合并旧日志作为历史数据。
6. 验证新流程稳定后，再考虑把旧日志创建入口迁移到作业单创建入口。

## Open Questions

- MVP 的人工成本来源关联是新增 `source_type/source_id` 字段，还是先用单独关联表连接作业单和账单。
- 种植单元状态枚举是否只保留 `active/paused/finished`，还是需要支持“空棚”“待定植”。
- 西瓜作业模板是否需要按地区或露天/大棚区分默认字段提示，MVP 可先不区分。
