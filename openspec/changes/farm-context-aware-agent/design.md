## Context

当前系统 Agent 的回复是"通用型"的：system prompt 硬编码"擅长西瓜、豆角"，Agent 不知道用户农场当前实际在种什么。用户创建了一个"春季西瓜"茬口后，每日建议 API `/agent/daily` 如果不传 `cycle_id`，Agent 的 prompt 里没有任何关于当前茬口的信息，只能泛泛而谈。

同时，农业中的赊账场景（向农资店赊购大棚膜、雇工工资月结、买家先拿货后付款）在系统中完全缺失。成本记账只有 `cost` 和 `income` 两种类型，没有"欠钱"和"被欠钱"的状态跟踪。

系统约束不变：2h2g 服务器、SQLite、3-4 个农户用户。

## Goals / Non-Goals

**Goals:**
- Agent 能感知农场当前所有活跃茬口，按作物分别生成建议
- 成本记账支持赊账/应收应付，支持记录债务人/债权人、还款状态
- 用户可自由创建作物模板（小瓜、辣椒等），替代硬编码西瓜+豆角
- Agent 调用前自动注入农场上下文摘要（茬口、成本、债务、天气）
- 移动端支持赊账录入和多作物建议展示

**Non-Goals:**
- 真正的向量 RAG（2h2g 不跑向量库）
- 自动还款提醒推送（无推送通道）
- 多农场管理（MVP 只支持单农场多作物）
- 作物病虫害图片识别
- 农资店/买家作为独立用户角色（只记录名称，不做用户体系）

## Decisions

### Decision 1: 赊账数据模型扩展 CostRecord

**选择**: 在现有 `cost_records` 表中扩展字段来支持赊账，而非新建独立表。

**新增字段**:
- `record_subtype`: `direct`(现结) / `debt`(应付) / `receivable`(应收)
- `counterparty`: 对方名称（农资店名、雇工名、买家名）
- `due_date`: 约定还款/收款日期
- `settled_at`: 实际结清日期（NULL 表示未结清）
- `parent_record_id`: 关联的原始记录（还款时指向原赊账记录）

**理由**:
- 赊账本质仍是成本/收入，只是结算时间不同，同一张表查询更简单
- 现有成本统计 API 只需增加 `record_subtype` 过滤即可兼容

**替代方案**: 独立 `debt_records` 表。拒绝理由：增加 JOIN 复杂度，统计成本时需要 UNION 两张表。

### Decision 2: Agent 上下文注入采用「结构化摘要」而非 RAG

**选择**: 每次 Agent 调用前，后端自动查询数据库组装一段「农场现状摘要」文本，直接注入 prompt 的 `{{ farm_context_summary }}` 变量。

**摘要内容**:
```
【当前农场状态】
- 活跃茬口：春季西瓜（伸蔓期，预计6/15采收）、秋季豆角（播种期，预计7/1开始）
- 近期农事：5/20施肥，5/22浇水
- 本月成本：3200元（化肥2000，人工1200）
- 应收应付：欠农资店3000元（大棚膜，约定6/30还），雇工老王1500元（约定月底结）
- 未来天气：未来3天有雨，注意排水
```

**理由**:
- 用户需要的信息全是结构化数据（茬口状态、成本金额、债务关系），用 SQL 查询比向量检索更精准
- 2h2g 服务器无法运行向量数据库，SQLite FTS5 对数值查询也不友好
- 摘要长度可控（约 300-500 字），不会显著增加 token 成本

**替代方案**: 引入轻量 RAG（SQLite FTS5 + 文本分块）。拒绝理由：FTS5 对"欠多少钱"这类聚合查询无能为力，还是需要 SQL。

### Decision 3: 茬口状态自动计算

**选择**: `CropCycle.status` 不再由用户手动维护，而是根据阶段日期自动计算。

**规则**:
- `active`: 存在某个阶段的起止日期覆盖今天
- `upcoming`: 所有阶段的开始日期都在未来
- `completed`: 所有阶段的结束日期都在过去
- `paused`: 用户手动标记暂停（保留手动干预能力）

**理由**:
- 减少用户维护负担，避免忘记更新状态导致建议错位
- 每日建议查询时只需 `status='active'`，逻辑清晰

### Decision 4: 多作物建议按卡片分组展示

**选择**: 移动端每日建议页从"单条建议文本"改为"多卡片布局"，每个 active 茬口一个卡片。

**理由**:
- 用户可能同时种西瓜和豆角，一条混在一起的建议难以阅读
- 卡片化后用户可以快速滑动浏览不同作物的建议
- 如果只有一个 active 茬口，自动折叠为单卡片，不增加复杂度

### Decision 5: 作物模板从系统预设改为用户自定义

**选择**: `CropTemplate` 表已存在，但当前只预置了西瓜和豆角。改造为：
1. 保留系统预设模板作为"示例"
2. 用户可在移动端创建自定义模板（名称、品种、各阶段名称和天数）
3. 创建茬口时选择任意模板（系统或自定义）

**理由**:
- 已有 `CropTemplate` 模型，改造成本低
- 用户秋季可能种小瓜、辣椒等，不能写死在代码里

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 上下文摘要过长导致 token 爆炸 | 限制摘要长度（≤500 字），债务只展示未结清的，天气只展示未来3天 |
| 多茬口并行时建议质量下降 | 每个茬口独立生成建议后合并，而非一次性生成所有建议 |
| 赊账和现结混在一张表导致查询混乱 | `record_subtype` 字段强制过滤，API 默认只返回 `direct` |
| 用户创建的作物模板质量不一 | 提供系统预设模板作为参考，不强制校验阶段科学性 |
| 自动计算的茬口状态与用户预期不符 | 保留 `paused` 手动状态，允许用户覆盖自动计算 |

## Migration Plan

1. **数据库迁移**：新增 `cost_records` 字段（`record_subtype`, `counterparty`, `due_date`, `settled_at`, `parent_record_id`）
2. **数据回填**：现有记录的 `record_subtype` 默认为 `direct`
3. **后端部署**：新 API 兼容旧客户端（不传新字段时按原逻辑处理）
4. **移动端更新**：发版后支持赊账录入和多卡片建议
5. **Agent Prompt 更新**：待 prompt-governance 完成后，在 `base.j2` 中加入 `{{ farm_context_summary }}` 占位符

**Rollback**: 如果新字段导致问题，旧客户端仍可正常使用（新字段有默认值）。

## Open Questions

1. 赊账的"还款"操作是创建一条新记录（parent_record_id 指向原记录）还是修改原记录的 `settled_at`？（倾向创建新记录，保留历史）
2. 上下文摘要是否包含"上周同期做了什么"这类对比信息？（MVP 先不包含，后续扩展）
3. 多作物建议是每个茬口独立调用一次 LLM，还是一次调用生成所有建议后拆分？（倾向一次调用，减少 LLM 成本）
