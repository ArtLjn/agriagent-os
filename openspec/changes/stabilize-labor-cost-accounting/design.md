## Context

当前系统已经有种植批次、作业单、工人、用工明细和账单模型，但移动端录入流程还没有把“工人工资 -> 人工成本账单 -> 利润总支出”做成一个稳定闭环。用户希望首页有全局工人管理入口，工人可以跨多个茬口工作；记工资时可绑定茬口、作物、作业，并通过下拉选择减少手输错误。账单页和利润页必须共享同一套数据口径，不能因为人工来自作业单或工资页而漏算、重算。

## Goals / Non-Goals

**Goals:**
- 建立全局工人管理、记工资、茬口用工摘要、账单支出、利润统计之间的一致数据流。
- 记工资页使用下拉选择、搜索选择、快捷人数和快捷单价控件，降低录入错误。
- 工资保存后生成唯一人工成本账单，并能从账单页追溯到用工记录、工人和茬口。
- 利润统计的总支出必须等于同一筛选口径下账单页支出合计。
- 茬口详情只展示本茬用工摘要，不把工人档案固定归属到某个茬口。

**Non-Goals:**
- 不做完整人事管理，如身份证、合同、考勤排班、绩效。
- 不做复杂工资规则，如计件阶梯价、社保、个税。
- 不引入新的会计科目体系，继续复用现有账单分类与成本记录。

## Decisions

1. 工人作为全局档案，工资记录绑定业务上下文。
   - 选择：`workers` 是全局农场资源；`labor_entries` 绑定 `work_order_id`，并通过作业单关联 `cycle_id`，独立记工资时创建轻量工资作业单或直接绑定 `cycle_id` 的工资上下文。
   - 理由：同一个工人会跨西瓜、豆角等多个茬口工作，不能把工人放进某个茬口作为固定子资源。
   - 替代方案：把工人挂在茬口下。放弃原因是跨茬口工资统计、欠款汇总和工人复用会变复杂。

2. 人工成本账单由工资保存流程统一生成或更新。
   - 选择：工资保存后写入 `labor_entries`，再生成一条 `cost_records`，`source_type` 使用稳定枚举，如 `labor_entry` 或 `operation_work_order`，`source_id` 指向来源。
   - 理由：账单页、利润页和报表都以 `cost_records` 为财务事实来源，避免人工成本散落在多个表里。
   - 替代方案：利润统计直接读取 `labor_entries`。放弃原因是会绕过账单页，导致总支出和账单流水不一致。

3. 幂等以来源键控制，禁止重复人工成本。
   - 选择：同一来源的人工成本账单必须唯一；重复保存或重试时更新原账单，而不是插入新账单。
   - 理由：移动端网络重试、用户重复点击保存都可能导致重复成本，必须在服务层和数据库约束层防御。
   - 替代方案：只在前端禁用按钮。放弃原因是无法防住 API 重放和异常重试。

4. 记工资页优先选择控件，备注才允许自由文本。
   - 选择：茬口、作物、作业、工人使用下拉/搜索选择；人数和单价提供快捷 chip；备注支持自然语言补充。
   - 理由：财务数据需要结构化，不能依赖用户在备注里写“老王 200”。

5. 利润页与账单页通过相同查询条件聚合。
   - 选择：利润接口使用成本服务的同一套过滤条件：farm、cycle、record_type、category/source，账单页点击筛选和利润页聚合共用 query contract。
   - 理由：保证“利润总支出”与账单页筛选后的支出合计一致。

## Risks / Trade-offs

- [Risk] 既有作业单人工成本已经生成过，新增幂等规则可能遇到历史重复数据。→ Migration 前增加重复来源扫描，发现重复项先报告，不自动删除。
- [Risk] 独立记工资没有具体作业单时，来源追溯不清晰。→ 使用“工资记录”作为来源类型，并要求至少绑定茬口，作业可选。
- [Risk] 工人工资修改后账单金额不同步。→ 工资更新必须通过同一服务方法，重算 labor_entries 汇总并更新关联 cost_records。
- [Risk] 前端下拉数据过多影响操作。→ 工人选择使用搜索选择，默认展示最近使用工人；茬口默认选当前上下文。
- [Risk] 总支出口径被多个页面各自实现。→ 把聚合逻辑集中在后端成本/利润服务，前端只消费结果和筛选参数。

## Migration Plan

1. 增加迁移前审计命令，扫描 `cost_records.source_type/source_id` 重复人工来源和未关联人工账单。
2. 补充必要的唯一约束或服务层幂等检查。
3. 后端先发布兼容 API，保持旧作业单创建人工成本逻辑可用。
4. 移动端切换记工资和工人管理入口。
5. 发布后运行利润总支出与账单支出一致性检查。
6. 回滚时保留新增字段和数据，前端隐藏新入口，后端继续兼容旧账单读取。
